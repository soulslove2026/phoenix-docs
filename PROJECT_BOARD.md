## 8. Project Rules

1. No production code before the engineering foundation is approved.
2. No critical business rule may exist only in conversation.
3. Every major decision must be documented.
4. Every release must update the changelog and master index.
5. No domain may directly modify another domain's database.
6. No financial balance may be changed without ledger records.
7. No third-party service may be embedded without an adapter and evaluation.
8. No AI-generated implementation may bypass human review for finance, security, privacy, moderation, or identity.
9. Arabic RTL and English LTR are mandatory from the first production release.
10. Phoenix must preserve an original identity and must not copy protected assets or proprietary designs.

---

## 9. Current Risks

| Risk | Severity | Response |
|---|---:|---|
| Premature coding before requirements stabilize | High | Block production implementation |
| Documentation becoming fragmented | High | Use master index and navigation hubs |
| Overengineering too early | Medium | Modular monolith first |
| Financial ambiguity | High | Build ledger and payment rules before implementation |
| AI-generated inconsistencies | High | Enforce AI context and coding protocol |
| Repository upload mistakes | Medium | Use versioned ZIP releases and release notes |
| Scope expansion | High | Prioritize phased roadmap and exit criteria |
